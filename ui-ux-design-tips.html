Good user experience (UX) is not superficial — it is functional. Every design decision impacts how the user feels, how long it takes to complete a task, and whether they come back. The tips below were extracted from the "Building with Good UX" and organized into 6 practical categories.

1

Loading States

Choose the right loading for each context
Problem: The user has to wait. Every action in software has latency — network, processing, database. Without visual feedback, the user does not know if something is happening or if the app crashed. The consequence? Frustration, abandonment, and loss of trust.

Different operations communicate different wait times. An infinite spinner in a 30-second process creates anxiety. A progress bar that does not reflect real progress breaks trust. The choice of loading component is a visual message about how much time is left.

Type When to use Visual feedback
Spinner inline Fast Short actions (<3s): save, update, submit Spins near the element, does not block the screen
Skeleton screen Medium Structural content loading (feed, list) Pre-draws the layout, reduces perceived wait
Progress bar Long Processes with predictable duration (upload, download) Percentage or steps — shows how much is left
Full-screen spinner ⚠ Blocking Full screen transitions (login, initial load) Blocks interaction — use sparingly
💡 Key principle: The loading type communicates the expected duration. Choosing the wrong type frustrates more than the wait itself.
2

Forms

6 tips for one of the most hated (yet necessary) software interactions
Problem: Forms are the "necessary evil" of software development. They are the main barrier between the user and their goal — signup, purchase, contact. A poorly designed form causes abandonment, input errors, frustration, and unnecessary support.

The good news: small changes have enormous impact. Each micro-interaction is an opportunity to guide, reassure, and reward the user. The 6 tips below form a practical checklist for any form.

✓

Input Feedback

Problem: User does not know if they filled correctly until they try to submit.

Solution: Immediate visual feedback — green border for valid, red for error, specific message next to the field. User corrects on the spot, not at the end.

👁

Show/Hide Password

Problem: Hidden passwords cause typing errors (especially on mobile). User needs to see what they typed.

Solution: Eye icon to toggle visibility. Reduces login errors by up to 30% according to UX studies.

⚖️

Disabled vs Enabled

Problem: Disabled (gray) button does not tell the user why they cannot submit. They click, nothing happens, frustration.

Solution: Enabled button with inline validation. Shows specific errors in each field. User sees what needs fixing.

✨

Micro-interactions

Problem: Dry transitions and trivial feedback make the form mechanical and impersonal.

Solution: Small animations (checkmark on validate, shake on error, button that "sinks" on click) turn the task into a pleasant experience.

#

Input Formatting

Problem: SSN, phone, date — each format has rules. User gets punctuation, spaces, digits wrong.

Solution: Auto masks while typing (e.g. (11) 99999-0000). Reduces format errors by ~60% and speeds up filling.

⚠

Error Handling

Problem: Generic messages ("Server error") or backend validation without clear feedback.

Solution: Specific messages next to the field in clear language. E.g. "Invalid email — use format name@domain.com" instead of "Invalid field".

💡 Key principle: Every micro-interaction in a form is an opportunity to guide, reassure, and reward the user. Small changes → big impact on conversion rate.
3

Empty States

The first impression your user has of your app
Problem: Developers and designers spend weeks testing the app with real data. But the new user arrives at an empty app — no items, no posts, no history. A generic blank screen is confusing, impersonal, and increases the chance of abandonment in the first session.

The empty state is not a "problem to be solved" — it is an onboarding opportunity. It is the user first conversation with your product. A well-designed empty state educates, engages, and guides the user toward the desired action.

  • ✓ Do not ignore it: Developers test with data. Users arrive empty. Plan the empty state as part of the flow.
  • ✓ Be memorable: Use illustration, brand tone, welcoming message. A blank screen is a missed opportunity.
  • ✓ Give action: Clear call-to-action — "Add your first item", "Connect your account", "Create your first project".
  • ✓ Natural onboarding: A well-crafted empty state educates the user about the app purpose in practice, not with a boring tutorial.
💡 Key principle: The empty state is the first digital impression of your app. Treat it as an onboarding opportunity, not a technical problem.
4

Partial States

When part of the content is ready and part is not
Problem: Not every load is binary (all or nothing). In modern apps, data comes from multiple sources — API, local cache, CDN, database. Waiting for everything to load before showing anything degrades the experience. The user stares at a spinner while already-available content could be consumed.

Partial states are the realistic approach for connected apps. Instead of blocking the screen until everything loads, show what is already available while the rest arrives.

  • ✓ Gradually loading feed: Show what is ready while the rest loads. User starts interacting immediately.
  • ✓ Images with placeholder: Background loads, image appears as it downloads. Do not hold text waiting for the photo.
  • ✓ Priority data first: Load critical content, then secondary. Users prefer reading text without an image over staring at a spinner.
💡 Key principle: Show something useful as fast as possible. Users prefer interacting with partial content over waiting for everything.
5

Graceful Degradation

When something fails, the app keeps working (in reduced form)
Problem: In software, failures happen — network goes down, API returns error, third-party service is out. The naive approach is to show a generic error and block the user. The result: a frustrated user who cannot use the app for a problem that is not theirs.

Graceful degradation is the art of failing with elegance. When a feature breaks, the app keeps operating in reduced mode — dependent features are gracefully disabled, and the user is informed clearly and friendly.

  • ✓ Offline mode: Show cached data instead of error screen. User stays productive even without network.
  • ✓ Reduced functionality: Disable features that need network, but do not break the whole app. E.g. "Comments unavailable offline" instead of a 404 page.
  • ✓ Friendly messages: "You are offline. Your changes will sync when connection returns" is better than "Error 500".
💡 Key principle: Planning for failure is as important as planning for success. Users prefer partial functionality over total blockage.
6

Consistency & Authenticity

The human factor in digital product design
Problem: Digital products are built by people, for people. Delays, unforeseen events, and plan changes are inevitable. How you communicate with your users during these moments defines trust in your product.

Part 10 of the series did not bring a technical UI/UX tip, but a valuable lesson about the reality of content and software production: maintain your standard, be transparent about unforeseen events, and honor your commitments — even if it means pauses in publishing frequency.

  • ✓ Schedule transparency: If something is delayed, communicate. Silence creates anxiety; an honest update builds trust.
  • ✓ Quality > frequency: It is better to pause and deliver something well-made than to publish half-baked content/features just to meet a calendar.
  • ✓ Be human: Unforeseen events happen (travel, illness, technical issues). Treating users as partners — not numbers — builds loyalty.
💡 Key principle: Transparency builds trust. An honest communication about limitations is worth more than a perfect but silent product.
